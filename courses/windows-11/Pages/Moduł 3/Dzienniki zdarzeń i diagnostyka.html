Dzienniki zdarzeń i diagnostyka

Gdy komputer się zawiesza, usługa nie startuje, albo użytkownik twierdzi że „samo się wyłączyło" – Twoja jedyna szansa na ustalenie przyczyny to Dziennik zdarzeń (Event Viewer). Windows rejestruje praktycznie wszystko, co się dzieje w systemie – od pomyślnego logowania po krytyczną awarię jądra. Wystarczy wiedzieć, gdzie szukać.

📖 Podgląd zdarzeń (`eventvwr.msc`)

+ R → eventvwr.msc

Struktura dzienników

Okno po lewej stronie pokazuje drzewo folderów. Najważniejsze wzruty:

Dzienniki Windows Dzienniki aplikacji i usług Widoki niestandardowe

Dzienniki Windows – główne źródło informacji

Dziennik Co rejestruje Kiedy zaglądasz
Aplikacja Zdarzenia z programów (błędy, ostrzeżenia, informacje) Crash aplikacji, błędy Office, awarie programów
Zabezpieczenia Logowania, zmiany uprawnień, audyt dostępu Kto się logował? Czy ktoś próbował włamać się do konta?
System Zdarzenia systemu operacyjnego (sterowniki, usługi, hardware) BSOD, awarie sterowników, problemy z uruchomieniem
Konfiguracja Instalacja aktualizacji i komponentów Windows Dlaczego update się nie zainstalował?

Na Helpdesku w $90%$ przypadków wystarczą Ci dwa dzienniki: System (sprzęt i usługi) i Aplikacja (programy).

Dzienniki aplikacji i usług

Zagnieżdżone foldery, z których najważniejszy to:

Microsoft → Windows → tu znajdziesz setki podsystemów:

  • Diagnostics-Performance – wolne uruchamianie systemu
  • TaskScheduler – błędy Harmonogramu zadań
  • WindowsUpdateClient – szczegóły aktualizacji
  • WLAN-AutoConfig – problemy z Wi‑Fi

✨ Wskazówka

Nie musisz znać wszystkich poddzienników. Gdy szukasz problemu z konkretnym komponentem – wejdź do odpowiedniego folderu i posortuj po dacie. Zdarzenia błędów/krytyczne będą w kolorze czerwonym.

Widoki niestandardowe – Twoja tablica filtrów

Zdarzenia administracyjne – gotowy widok, który zbiera tylko błędy i ostrzeżenia ze wszystkich dzienników w jedno miejsce. To Twój pierwszy przystanek przy diagnostyce.

Możesz też tworzyć własne widoki – np. filtr pokazujący wyłącznie krytyczne błędy z ostatnich $24$ godzin:

  1. PPM na Widoki niestandardowe → Utwórz widok niestandardowy
  2. Zalogowane: Ostatnie $24$ godziny
  3. Poziom zdarzenia: Krytyczny Błąd
  4. OK → gotowy filtr do szybkiej diagnostyki

🧰 Kluczowe Event ID dla Helpdesku

Zapamiętaj te numery – pozwolą Ci natychmiast zidentyfikować przyczynę problemu:

Event ID Źródło Co oznacza Akcja
41 Kernel‑Power Nieoczekiwany restart / utrata zasilania (tzw. Kernel Power 41) Sprawdź zasilacz, kabel, UPS. Częste na laptopach z wadliwą baterią.
1000 Application Error Crash aplikacji (podaje nazwę .exe i moduł, który zawinił) Zaktualizuj lub przeinstaluj program. Sprawdź kompatybilność.
1001 Windows Error Reporting Raport błędu po crashu – szczegóły BugCheck (BSOD) Przeczytaj kod stopu – wskazuje winny sterownik.
6008 EventLog Poprzednie zamknięcie systemu było nieoczekiwane Potwierdza nagłe wyłączenie (prąd, crash, przytrzymanie przycisku).
7036 Service Control Manager Usługa zmieniła stan (uruchomiona / zatrzymana) Diagnostyka problemów z usługami (np. Spooler się zatrzymał).
7031 Service Control Manager Usługa nieoczekiwanie się zakończyła (crash usługi) Sprawdź zakładkę Odzyskiwanie w właściwościach usługi.
4624 Security Pomyślne logowanie Audyt – kto i kiedy się logował.
4625 Security Nieudane logowanie Próba włamywania? Złe hasło? Zablokowane konto?
10016 DCOM Błąd uprawnień DCOM (występuje powszechnie) W $99%$ przypadków ignoruj – znany, nieszkodliwy bug Windows.

✨ Wskazówka

Helpdeskowy trik: Gdy widzisz Event ID, którego nie znasz – skopiuj numer i źródło, wklej w wyszukiwarkę (np. „Event ID 41 Kernel‑Power"). Microsoft i społeczność od lat dokumentują rozwiązania dla każdego kodu.

⌨️ Dzienniki z linii poleceń

CMD (wevtutil):

wevtutil qe System /c:10 /f:text /rd:true    ← ostatnie 10 zdarzeń systemowych
wevtutil qe Application /c:5 /f:text /rd:true ← ostatnie 5 zdarzeń aplikacji
wevtutil cl System                             ← wyczyść dziennik System (ostrożnie!)

PowerShell (Get-WinEvent):

Get-WinEvent -LogName System -MaxEvents 10                     # ostatnie 10 zdarzeń
Get-WinEvent -LogName System | Where-Object {$_.LevelDisplayName -eq "Error"} | Select-Object -First 5
                                                               # ostatnie 5 błędów
Get-WinEvent -FilterHashtable @{LogName='System'; ID=41}       # tylko Event ID 41

🕵️ Scenariusze diagnostyczne

W dzienniku System widzisz powtarzający się wpis: Event ID 41, źródło Kernel‑Power, poziom Krytyczny, opis: „System został ponownie uruchomiony bez wcześniejszego czystego zamknięcia." Klient twierdzi, że nie wyłączał komputera. Co to oznacza i co sprawdzasz w pierwszej kolejności?

Wirus zainfekował jądro systemu i wymusza restarty – trzeba sformatować dysk.

Windows Update instaluje się w tle i automatycznie restartuje komputer.

Kernel‑Power 41 oznacza, że system stracił zasilanie lub zawiózł się na poziomie jądra. Sprawdź zasilacz, kabel, akumulator (laptop), temperatury procesora i karty graficznej. Sprawdź też BugCheck code – jeśli jest niezerowy, to BSOD.

To normalne zachowanie diagnostyczne Windows – system okresowo się restartuje by oczyścić pamięć RAM.

Kernel‑Power 41 to NIE normalne zachowanie i NIE jest automatycznie winą wirusa. To jeden z najpoważniejszych komunikatów – oznacza utratę zasilania lub twardę awarię. Przyczyny są zwykle sprzętowe.

Świetnie! Kernel‑Power 41 to „złoty standard" diagnostyki nagłych wyłączeń. Pole BugCheck w szczegółach zdarzenia powie Ci czy to był BSOD (niezerowy kod) czy utrata prądu (zerowy). To rozróżnienie kieruje Cię albo na problemy ze sterownikami (BSOD) albo na problemy z zasilaniem (hardware).

✨ Wskazówka

Złota zasada technika: Podgląd zdarzeń to Twoja „czarna skrzynka". Gdy klient mówi „nie wiem co się stało" – Event Viewer wie. Zacznij od Widoki niestandardowe → Zdarzenia administracyjne i przefiltruj po czasie wystąpienia problemu.
